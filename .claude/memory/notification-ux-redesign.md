**request_access grant bugs (2026-06-02, live-debugged against ~/gantry runtime).** Two real bugs in the request_access capability approval flow (branch code, not notification work):
1. "Always Allow" → "Not approved: ... Reason: persistent rule allowed via Telegram". Root cause: startRequestOnlyCapabilityReview (jobs/ipc-admin-handlers.ts) built the PermissionApprovalRequest WITHOUT semanticCapabilityDefinitions. The button-eligibility (requestPermissionSetupDecisionOptions) doesn't validate the definition, but decisionForMode→persistentPermissionUpdates→persistentRuleForSuggestion→validateDurableAccessRule REQUIRES request.semanticCapabilityDefinitions[capabilityId] for a `capability:<id>` rule (else "Unknown semantic capability" → approved:false). Telegram (channel-connect.ts:281) then overrides decision.reason regardless of approved, producing the contradictory ack. Fix: new resolveTrustedSemanticCapabilityDefinitions(deps,appId,agentId) (catalog via exported semanticCapabilityDefinitionsFromToolCatalog + skillActionDefinitionsForAgent) attached to the request. Same missing-definition cause also crashed STARTUP reconciliation ("Unknown semantic capability google.sheets.values.get" in desired-state-capability-reconcile) when a stale grant was in settings.yaml.
2. "Allow" (once) on a capability approved but added NO live tool rule (requestPermissionTransientLiveRules returns [] for capabilityId) → agent told approved but couldn't use it that run. Fix: requestPermissionOnceLiveRules expands capability via expandSemanticCapabilityPermissionRules → appends `capability:<id>` + runtime rules (e.g. RunCommand(/opt/homebrew/bin/gog sheets get *)) as live rules. "Allow 5 min" is NOT offered for capabilities (decisionOptions = [allow_once, allow_persistent_rule, cancel]) — by design.

Runtime ops: ~/gantry runs `dist/index.js` as launchd service `com.gantry`; restart in place with `launchctl kickstart -k gui/$(id -u)/com.gantry` (picks up a fresh `npm run build`). A running node process does NOT hot-reload dist — must restart.
