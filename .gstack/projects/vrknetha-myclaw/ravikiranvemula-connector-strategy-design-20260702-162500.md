# Design: Gantry Product-Level Connector Strategy (org-owned OAuth + API-key connections, granular tool grants)

Generated by /office-hours on 2026-07-02

## Problem Statement

Gantry has fully-built governed capability rails — sources vs semantic capabilities,
per-agent bindings with `allowedToolPatterns`, `credentialRefs` → AES-256-GCM encrypted
capability secrets, agent request → in-conversation admin approval → settings-version
lifecycle, and a lazy `mcp_list_tools` / `mcp_describe_tool` / `mcp_call_tool` meta-tool
surface. What it does not have is any third-party OAuth: every integration today is a
pasted token or API key, with no refresh and no revocation story.

The founder wants a product-level connector strategy: the super admin of each Gantry
deployment connects external accounts (OAuth where the provider supports it, API key where
not) from any surface (UI/app/CLI), assigns connectors to agents with granular per-tool
access, agents request additional tools in conversation and admins approve there, and
connector tools reach agents only through the existing smart search/execute surface —
never bulk-projected.

## Demand Evidence

Honest framing: this is a **sales-demo and internal-dogfood driver, not a blocked
customer**. Founder verbatim: "We need the bare minimum connectors like Google, Jira,
GitHub so that even internally it would be easy to use and to show it to customers.
Showing list of connectors which are famous for all orgs helps easy to sell Gantry as we
will be giving demos." KnackLabs has two paid enterprise engagements (pre-launch); buyer
conversations expect a recognizable integration gallery. No customer is currently blocked
on a specific connector — the first "user" is KnackLabs itself.

## Status Quo

Admins paste API keys via CLI (`gantry mcp connect --credential ...`, capability-secret
store) and bot tokens for channels. Consequences: OAuth-only providers — most of Google
Workspace — are unreachable; tokens never refresh or revoke; every integration is manual
key wrangling per deployment. Agents already avoid tool bombardment via the MCP meta-tool
surface, so the discovery/execution half of the problem is solved; the connection half is
not.

## Target User & Narrowest Wedge

- **User:** the super admin of each Gantry deployment — KnackLabs staff today (internal
  gantry + customer builds), customer IT admins later.
- **Wedge (founder-set):** GitHub **and** Google connectors end-to-end — connect, assign,
  search, request, approve, execute, audit, revoke — plus a static connector **catalog**
  where famous providers (Jira/Atlassian next) show as "coming soon." Depth on two, logos
  for the rest. Founder verbatim: "We need the plumbing needed so that we start
  integrating it one by one."

## Constraints

1. **No new deployable infrastructure per customer install** — the founder rejected
   embedding self-hosted Nango for this reason. Lean enterprise installs win.
2. Reuse existing rails wholesale: capability model, approval flow, audit, secrets crypto,
   MCP source/binding stack, control API, CLI.
3. Connector tools flow only through the search/execute meta-tools regardless of connector
   count.
4. Org-owned connections in v1; schema must be keyed so per-user connections land later
   without migration.
5. v1 OAuth apps are KnackLabs-registered internal/test-mode apps (demo + dogfood scope);
   per-customer OAuth app registration and Google scope verification are deferred.
6. **(revised 2026-07-02, founder)** A proper Connectors web UI is v1 scope: non-developer
   admins must see the integration gallery, connect accounts, and manage agent access from
   a browser — CLI/programmatic MCP registration is not an acceptable admin path. The UI
   stays one self-contained page served by the existing control API, not a dashboard
   build-out.

## Premises (agreed 2026-07-02)

1. A connector = connection (auth) + curated tool surface + granular per-tool grants; the
   new build is the connection registry, browser OAuth flow, token refresh/resolution,
   curated toolpacks, and a catalog surface. The capability/approval/audit rails
   themselves are not rebuilt.
2. v1 connections are org-owned and admin-created, keyed (`ownerType: 'org' | 'user'`) for
   per-user later.
3. Connector tools are never bulk-projected onto agents.
4. **(revised after cross-model challenge)** v1 = GitHub + Google end-to-end behind a
   uniform provider interface; Jira and others ship as coming-soon catalog entries and are
   added one by one on the same rails.
5. v1 OAuth apps are internal/dev registrations; production per-customer OAuth apps and
   Google verification are explicitly deferred scope.
6. **(revised after founder correction)** v1 ships a Connectors web UI served by the
   control API — gallery (connected / available / coming-soon, with logos),
   connect/disconnect, and agent assignment with per-tool checklists — usable by
   non-developer admins. Entry is a signed magic link handed out from chat/CLI/API; no
   separate login system is built in v1.
7. Do-nothing cost is real: API-key paste excludes OAuth-only providers and has no
   refresh/revocation story.

## Approaches Considered

### Approach A: Direct OAuth, no new infra (CHOSEN, with Nango-derived provider templates)
Build OAuth inside Gantry: authorize/callback routes on the existing control API, token
exchange, refresh, tokens in the existing encrypted capability-secret lane. Founder
refinement: provider-specific behavior lives in declarative template blocks ported from
Nango's `providers.yaml` (ELv2 permits this) driving one generic engine, so new providers
are config + toolpack, not code. Founder correction (post-review): a proper Connectors
web UI for non-developer admins is v1 scope, served as static assets from the control
API. Effort M-L (human ~3-4 wks / CC ~5-8 days; the refresh-aware materialization seam,
the Google toolpack, and the web UI dominate). Risk Low-Med.
Pros: zero new deployable components; one credential trust boundary; gallery growth is
config-speed, not hand-rolled speed; non-dev admins get a real integrations surface.
Cons: Gantry owns refresh correctness forever (races, proactive refresh, revocation
detection); ported templates drift from upstream Nango fixes unless periodically
re-synced; connect page + first admin web UI built from scratch (moderate).

### Approach B: Embed self-hosted Nango as the credential plane (Claude + Codex recommendation, not chosen)
Run Nango free self-host (ELv2; Auth+Proxy tier: one container + Redis) beside each
Gantry; store only Nango connection references; ~817 provider auth configs; refresh and
Connect UI solved. Rejected by founder on the deployment-leanness constraint: one more
ops component (plus Redis, encryption-key management, upgrade cadence, under-documented
free/paid boundary) in every customer install.

### Approach C: Connectors as OAuth'd MCP sources (lateral, not chosen)
Lean on the MCP authorization spec against official remote MCP servers (GitHub, Atlassian).
Rejected as the backbone: spec young/uneven, Google and API-key providers don't fit, so
Approach A's machinery gets built anyway. Its best idea survives inside A: where an
official remote MCP server exists, use it as the tool surface.

## Recommended Approach

**Approach A — founder-chosen** (Claude recommended B for gallery-growth economics; the
founder's deployment context — lean per-customer enterprise installs — outvoted it). The
provider interface below is the deliberate seam that keeps a later Nango swap a contained
change, not a rewrite.

Build, reusing named existing seams:

1. **Connection registry.** New `connector_connections` table:
   `{id, provider, ownerType:'org' (userId nullable for later), status:
   connected|needs_attention|revoked, scopes, accountLabel, credentialSecretRef,
   oauthAppId, connectedBy, createdAt, lastRefreshAt, expiresAt}`. Modeled on the existing
   provider-accounts shape (`RuntimeProviderAccountSettings`,
   `apps/core/src/config/settings/runtime-settings-types.ts`). Tokens themselves live in
   the capability-secret lane (`capability_secrets`, AES-256-GCM,
   `credential-secret-crypto.ts`), referenced via `credentialSecretRef` — never inlined.
   `expiresAt` feeds the resolve-on-use expiry check. Agent bindings reference a specific
   `connectionId`. v1 enforces **one connection per provider per org** (unique
   constraint) so resolution is never ambiguous; the constraint is lifted — not migrated —
   when multi-account lands (`accountLabel` is already in place for it).
2. **Declarative provider pattern (founder directive: every next provider must be
   easy).** Do not hand-code a class per provider. Adopt Nango's `providers.yaml` template
   format as the provider registry: each provider is a **data block** (authorize URL,
   token URL, auth mode `OAUTH2|API_KEY|BASIC|OAUTH2_CC`, scope format, refresh quirks,
   token expiry hints) driving one generic engine that implements only `OAUTH2` in v1 —
   the format carries the other modes; each is implemented when its first provider ships.
   Port the format — and,
   provider by provider, the config blocks themselves — from Nango's repo
   (`packages/providers/providers.yaml`, ~817 entries). Adding provider #3..#N is then:
   import its YAML block, register an OAuth app, curate a toolpack. A thin
   `ConnectorProvider` escape hatch (`buildAuthorizeUrl`, `exchangeCode`, `refresh`,
   `revoke`) stays for genuinely custom flows, and doubles as the seam for
   a later full-Nango swap. Each provider also carries a static catalog descriptor (name,
   logo, auth kind, default scopes, tool pack id). v1 template entries: GitHub, Google.
   **Licensing note:** Nango is ELv2 — copying/adapting the config format, provider
   blocks, and reference logic into Gantry is permitted (Gantry is not offering Nango as a
   managed service and circumvents no license keys); preserve copyright notices on
   anything copied and record provenance in the file header. v1 imports only the field
   subset the OAUTH2 engine reads (authorize/token URLs, scope format, refresh/expiry
   hints); unread format fields are dropped, not carried as silent drift. **The
   config-only claim covers authorize/exchange/refresh.** Revocation is per-provider
   escape-hatch code from day one — Nango's format carries no revoke endpoints, and the
   v1 pair is non-uniform (GitHub: `DELETE /applications/{client_id}/grant` with Basic
   auth; Google: `POST /revoke`).
3. **OAuth flow on the existing control API** (`apps/core/src/control/server/`):
   `POST /v1/connectors/:provider/connect-sessions` → returns a short-lived signed URL;
   `GET /v1/connectors/oauth/callback` → validates `state`, exchanges code, encrypts and
   stores tokens, marks the connection connected. The connect-session URL is an
   HMAC-signed, single-use token bound to the session id and the initiating admin, TTL 15
   minutes; the callback validates it and correlates tokens back to that session. A
   minimal connect/confirmation page is served by the control API. CLI (`gantry
   connectors connect github`) and in-chat admin both just print the URL. Both v1
   providers are OAuth; the API-key/BASIC connect path (a paste form on the same connect
   page) is deferred until the first API-key provider ships, and the existing `gantry mcp
   connect --credential` paste path coexists unchanged (no migration in scope).
   Per-provider OAuth app config (client id, `oauthAppId`) lives in settings.yaml with
   the client secret in the capability-secret lane, like every other credentialRef.
   Re-running connect for an already-connected provider updates the existing connection
   row in place — same `connectionId`, agent bindings preserved. Deployment prerequisite:
   an externally reachable HTTPS callback URL per deployment (known exclusion:
   air-gapped/firewalled installs are out of v1 scope).
4. **Token resolution + refresh.** Resolve-on-use with expiry check; single-flight
   refresh via a Postgres advisory lock (or `SELECT ... FOR UPDATE`) on the connection
   row — Postgres is already shared across control API, scheduler, and runners, so no new
   coordination infra. Refresh failure flips status to `needs_attention` and notifies the
   deployment's admin conversation channel (same delivery as permission approvals). A
   proactive refresh job for long-idle connections is deferred — resolve-on-use covers
   correctness and idle decay doesn't matter for daily dogfood (re-entry path below).
   Provider-side revocation is detected at call time: a 401/`invalid_grant` from the
   provider (GitHub OAuth-app tokens never expire, so this is the only signal a user
   revoked in GitHub settings) flips the connection to `needs_attention` with the same
   admin notification. Gantry-side revocation (`gantry connectors disconnect` or the UI)
   calls `revoke()`, deletes secrets, and downstream tool calls fail closed with a clear
   reason; the same fail-closed rule is the integrity contract for the cross-store seam
   (bindings live in the settings lifecycle, connections in Postgres — a binding whose
   connection row is gone fails closed at call time, and disconnect warns which agents
   are still assigned). While a connection is `needs_attention`, tool calls still attempt
   resolve-on-use (a provider-side recovery heals without admin action) and return a
   clear error on failure; admins are notified **once per status transition, never per
   call**.
5. **Tool surface.** New `credentialSource: 'connector_connection'` in
   `semantic-capabilities.ts`. **Named engineering risk — design this seam first:** the
   existing `credentialRefs` materialization bakes credentials into MCP source config at
   spawn time, which breaks with rotating tokens. The decision: **in-process toolpacks
   resolve credentials per-call, core-side over the existing runner↔core IPC** — the
   runner never holds refresh tokens or `SECRET_ENCRYPTION_KEY`, only the short-lived
   access token for the call (trust boundary unchanged). The remote-MCP path needs **no
   reconnect-on-refresh in v1**: GitHub OAuth-app tokens don't expire, so v1 ships only
   the 401→`needs_attention` handling and disconnect teardown; reconnect-on-refresh is a
   re-entry path required before the first expiring-token remote-MCP provider. This
   materialization-seam change is deliberate, not pure reuse, and the riskiest item in
   the build. GitHub: prefer the official GitHub remote MCP server wired through the
   existing MCP source/binding rails; **first timeboxed step (1 day): verify it accepts
   our OAuth-app bearer token** — fall back to a curated REST toolpack if not. Google: curated in-process
   toolpack starting at 3-5 tools (Gmail search/read, Calendar list, Drive search); grow
   it from the dogfood request-log, not speculation. **Toolpack
   mechanism:** an in-process MCP source (same registration path as the existing internal
   Gantry tool surface) living in `apps/core`, so its tools appear in the same inventory
   `mcp_list_tools` already searches. **Scope-fit rule (in-process toolpacks only):**
   toolpack tools carry per-tool scope metadata and are curated to fit the connect-time
   OAuth scope set; a request for a tool needing a missing scope is answered "requires
   re-consent" and stays unavailable in v1 (re-consent is a re-entry path). The GitHub
   remote-MCP path is grant-pattern-only (`allowedToolPatterns`) — scopes are not checked
   per tool there. All tools surface through
   `mcp_list_tools`/`mcp_describe_tool`/`mcp_call_tool` only.
6. **Assignment + granularity + approval: zero new permission machinery.** Assigning a
   connector to an agent = existing binding + `allowedToolPatterns` / semantic-capability
   grants; agents request more via existing `request_access`; admins approve
   in-conversation via the existing Slack-button flow; audit via existing rails. (The UI
   writes these through one thin new REST wrapper — see item 8; the semantics are
   unchanged.)
7. **Catalog.** Static descriptor list (from the provider registry) exposed as
   `GET /v1/connectors/catalog` + `gantry connectors list`: available providers show
   connection state; famous-but-unbuilt providers (Jira first) show `coming_soon`. This is
   the demo gallery, shipped day one.
8. **Connectors web UI (founder directive: non-developer admins, no CLI in the loop).**
   One self-contained page of static assets bundled with core and served by the control
   API — no separate frontend service or framework build-out. Three views:
   **Gallery** — provider cards with logo and status (connected / available /
   coming-soon); Connect starts the OAuth dance in the same browser. **Connections** —
   per-connection card (account label, scopes, status incl. a `needs_attention` banner,
   Disconnect). **Access** — pick an agent, toggle connectors, per-tool checkboxes; saving
   writes exactly the same bindings/grants the CLI writes (no parallel permission path).
   **Entry/auth:** an HMAC-signed, single-use magic link (same mechanism as
   connect-sessions; minting requires an admin-scoped API key or in-chat admin identity;
   link TTL 15 minutes) minted from chat ("open connectors"), CLI, or REST. The link is
   exchanged for a **short-lived scoped API key held in page memory** and sent as the
   same `Authorization: Bearer` header every existing `/v1` route already authenticates —
   no cookies, no CSRF surface, no change to the shared `authenticate()`. Session TTL 60
   minutes; re-mint to continue. Durable admin login (SSO) is deferred — see Open
   Questions. **Data sources:** Gallery/Connections read the catalog and registry
   endpoints; the Access view reads the existing `/v1/inventory` for tool lists — a
   freshly connected remote-MCP provider whose tools haven't been inventoried yet shows a
   "listing tools…" state and triggers a refresh. **Write path:** one thin new REST
   wrapper (`GET/PUT /v1/agents/:id/connectors/:connectionId`) persisting the same
   bindings/grants through the same settings lifecycle the CLI uses — no new permission
   semantics; the endpoint pair is in scope here. **Implementation shape:** compiled to a
   single bundled static asset (framework optional); no separate deployable, no SSR
   service. This page is deliberately the seed of Gantry's admin web UI: keep it boring
   and self-contained so later dashboard work can absorb it.

**Explicit re-entry paths:** per-user connections (`ownerType`/`userId` already keyed);
Nango swap (implement `ConnectorProvider` against Nango's API when provider #6+ makes
hand-rolling dominate); per-customer OAuth apps (registry stores `oauthAppId` per
connection); scope re-consent flow (reconnect with expanded scopes when a requested tool
exceeds the connect-time grant); proactive refresh job for long-idle connections;
reconnect-on-refresh for remote-MCP sources (required before the first expiring-token
remote-MCP provider ships).

## Open Questions

1. **(decided during review)** Google v1 uses an *internal* OAuth app on the KnackLabs
   Workspace, and demos run against KnackLabs-owned demo accounts inside that org.
   External test-mode is a trap: its refresh tokens expire every 7 days, guaranteeing
   `needs_attention` churn in exactly the demo scenario. Remaining open: which GCP
   project owns the app.
2. GitHub tool surface: official remote MCP server vs small REST toolpack — MCP server
   preferred; the bearer-token verification spike is timeboxed to 1 day, then fall back
   to the REST toolpack without debate.
3. Trigger for building connector #3: first customer ask, or when Jira "coming soon" gets
   clicked N times in demos?
4. Revocation semantics mid-conversation: what exactly does a running agent see when an
   admin disconnects a connector it was using?
5. Encryption-key rotation for long-lived refresh tokens (existing keyring supports
   rotation — confirm the procedure covers the new secret kind).
6. Re-sync cadence for provider templates ported from Nango's `providers.yaml` (upstream
   fixes provider quirks continuously; decide pull-on-demand vs periodic diff).
7. Durable admin auth for the Connectors UI: magic link covers v1; decide when a real
   admin login (SSO/OIDC) becomes a customer requirement and whether it arrives with the
   broader dashboard.

## Success Criteria

The end-to-end lifecycle demo. Criteria 1-5 and 7 must pass on **both** providers; 6 is
Google-specific (short-lived access tokens); 8 is provider-independent:

1. A **non-developer admin** opens the Connectors UI via a magic link (from chat or
   handed over by ops), sees the gallery (GitHub, Google, Jira as coming-soon), clicks
   Connect on **the provider under test**, completes OAuth; the card shows connected with
   the account label. (`gantry connectors connect <provider>` remains an equivalent CLI
   path.)
2. The same admin assigns **that connector** to an agent with exactly two tools using the
   per-tool checklist in the UI — no terminal touched at any point in 1-2.
3. Agent discovers only those tools via `mcp_list_tools`; calls one successfully.
4. Agent requests a third tool via `request_access` (Google: within granted scopes per
   the scope-fit rule; GitHub: grant-pattern-only); admin approves with a Slack button;
   the tool works on the next turn.
5. Audit trail shows connection → grant → execution attributable end to end.
6. A Google access token past its 1-hour expiry refreshes automatically with no human
   action (verified by a >1h-idle then successful call).
7. Disconnect revokes at the provider, deletes secrets, and the agent's next call fails
   closed with a clear reason.
8. Catalog endpoint lists connected + coming-soon providers (the demo gallery).

## Distribution Plan

Ships inside Gantry core via the existing deployment pipeline; no new binaries, images, or
infra components. The only new deployment requirement is an externally reachable HTTPS
callback URL for the control API, documented in setup.

## Dependencies

### Provider prerequisites (v1 checklist)

**GitHub — OAuth App, not a GitHub App:**
- GitHub OAuth App under the KnackLabs org: `client_id` + `client_secret`; each
  deployment's callback URL in the allowlist.
- A dedicated machine user (e.g. `gantry-bot`) with access to the target repos — the
  admin authorizes as that account, making the connection org-owned rather than personal.
- Scopes: `repo`, `read:org`.
- No SDK required: token exchange via the generic engine; tools via the official remote
  MCP server (bearer) or REST fallback. A GitHub App (fine-grained per-repo permissions,
  JWT + private key + 1h installation tokens) is a named upgrade path, not v1.

**Google — GCP OAuth client, internal:**
- KnackLabs-owned GCP project, OAuth consent screen **Internal** (no verification, no
  7-day test-mode refresh expiry).
- OAuth 2.0 Client ID (Web application): `client_id` + `client_secret`, redirect URIs per
  deployment.
- Enable Gmail, Calendar, and Drive APIs in the project (off by default).
- Scopes (readonly v1): `gmail.readonly`, `calendar.readonly`, `drive.readonly`.
- Authorize with `access_type=offline&prompt=consent` to obtain the refresh token (1h
  access tokens; the resolve-on-use refresh engine handles rotation).
- No SDK required: standard OAuth2 token endpoints + direct REST for the 3-5 tool pack.

Landing spots: client IDs + `oauthAppId` in settings.yaml; client secrets in the
capability-secret lane; callback allowlist maintenance is the deployment dependency below.

### General

- GitHub OAuth app + Google Cloud OAuth consent (internal) registrations owned by
  KnackLabs (details above).
- Externally reachable control-API callback URL per deployment (dogfood gantry first;
  air-gapped installs explicitly out of v1 scope).
- Each deployment's callback URL must be added to the KnackLabs-owned OAuth apps'
  redirect-URI allowlists — manual at dogfood/demo scale; a central redirect proxy is the
  production-scale path, deferred.
- Existing `SECRET_ENCRYPTION_KEY` / capability-secret keyring in place.
- Connectors UI ships as static assets bundled with core (no separate frontend service,
  build pipeline addition only).
