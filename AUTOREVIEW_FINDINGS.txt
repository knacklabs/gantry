apps/core/src/adapters/llm/anthropic-claude-agent/gantry-model-gateway-routing.ts:197
Every ordinary OpenAI or Anthropic model-runtime token can now access batch submission, listing, and result-download paths because authorization checks only the provider, path, and method; the token's run scope is never considered. An agent holding its normal gateway token can therefore enqueue work outside `ChatBatchStateMachine`'s reservation/daily-cap accounting and inspect other batch results for the same credential. [P1] [P1] Preflight the provider payload before marking submission unknown

The intent is changed to `submission_unknown` before transport validation, credential resolution, or even the transport's abort check. Deterministic pre-send failures—such as missing credentials, an already-aborted signal, whitespace-only custom IDs, or a provider envelope exceeding the upload limit—therefore consume the daily reservation and remain unknown even though no request was sent. [P1] [P1] Preserve the output-token cap through durable submission
apps/core/src/memory/chat-batch-state-machine.ts:119
The batch port supports `maxOutputTokens`, and both transports honor it, but `ChatBatchSubmitInput` omits the field and this call never passes it. Consequently OpenAI batches omit `max_completion_tokens` entirely and Anthropic silently uses its hard-coded 4096-token default. That can invalidate the caller's cost reservation and make result files far larger than expected. Add the cap to the durable input, include it in the content hash/snapshot, and pass it to `submitBatch`.

[P1] [P1] Bound batch result downloads before buffering them
apps/core/src/adapters/llm/chat-batch-http.ts:49
`fetchBatchText` buffers the entire result file into one string; JSONL parsing then creates another line array and a parsed row array, while the OpenAI transport downloads output and error files concurrently. The 14 MiB upload limit does not bound output size: thousands of requests can each return thousands of tokens, so a valid completed batch can exhaust the core process repeatedly during result retries. Stream and parse JSONL incrementally with explicit byte/row limits before building the durable result snapshot.
