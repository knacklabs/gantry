| Customer-side image-bake CI pipeline | The sandboxed bake-job toolchain artifacts cover fleet correctness without customer-side CI. See [ADR Capability Artifacts](docs/decisions/2026-06-11-capability-artifacts.md). | A customer needs to build/publish their own worker images in their own CI. |
| Support-agent rate limiting | The existing conversation sender policy covers v1. | Observed abuse exceeds what the sender policy bounds, or a per-support-agent quota is required. |
| pip bake lane (live Python package installs in fleet) | The bake subsystem is npm-only by binding decision; the worker image now ships python3 so Python skill *scripts* run, but Python *packages* would need a pip lane (lockfile-pinned venv baked as an artifact, same shape as npm). Language enforcement lives in one place (the bake manifest validator), so this is a contained addition. | Skill authors need per-skill Python dependencies in fleet (not satisfiable by adding common libs to the image). |
| Pinned-binary bake lane (Go-built and other static CLI tools) | Go-built tools ship as static binaries, not toolchains; today each is one Dockerfile line (image roll). A bake lane distributing admin-approved URL + sha256-pinned binaries as verified artifacts would avoid image rolls per tool. The Go toolchain itself stays out of the image (runtime compilation = worker self-mutation). | Image rolls for individual CLI tools become frequent enough to hurt. |
| CLI dry-run probe with missing-OS-dependency report | When an agent requests a CLI tool absent from the fleet image, run the existing local_cli preflight as a dry run and return a structured "missing from worker image: <packages>" message the agent relays to the admin (who adds it to ops/docker/Dockerfile and rolls). Builds on the capability-paused one-clear-action surface; never triggers live installs on workers. | First fleet deployment where agents regularly request CLI tools not in the image. |
| Locked-agent unmet-need telemetry | Instruction stripping means a locked agent never attempts capability tools, so `denied_by_profile` audit rows capture forged attempts only — genuine customer demand ("N users asked for X this week") leaves zero signal. Needs a locked-safe demand record (audit event when the agent reports inability, or transcript mining) designed together with the human-handoff piece of the support product layer. | Locked support stack reaches production traffic, or the support product layer plan starts. |
| Live-conversation auto-resume after bake | Scheduled jobs self-heal when a capability lands (readiness unpauses → runs → notifies); a live conversation blocked on a dependency does not resume — the bake success notice tells the user to re-ask, which covers v1. Auto-resume would need a durable "blocked turn" record reattached on capability activation. | Users in fleet regularly hit the approve→wait→re-ask loop and the re-ask friction matters. |
