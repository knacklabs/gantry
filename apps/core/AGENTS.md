# Apps Core

## Scope

- `apps/core/src/` contains the runtime, routing, session, memory, and storage code for Gantry.

## Rules

- Keep runtime imports aligned with the split domains under `apps/core/src/` rather than rebuilding root wrapper modules.
- Service changes must keep `ops/bootstrap.sh`, `ops/launchd/com.gantry.plist`, and runtime diagnostics consistent.
- CLI onboarding code in `apps/core/src/cli/` must remain runtime-home based (`GANTRY_HOME`) and must not assume repo cwd.
- Keep prompt rendering separate from side-effect modules so onboarding behavior stays testable.
- `gantry` CLI commands should return actionable plain-English recovery guidance instead of raw startup failures.
- When path-sensitive code changes, update the matching tests in `apps/core/src/**/*.test.ts` in the same change.
- Integration tests for runtime features must use shared harnesses under `apps/core/test/harness/`; DB-backed cases must guard on `GANTRY_TEST_DATABASE_URL` and isolate schemas.
- Run `npm run test:integration:postgres` for DB-backed feature work. Use a disposable Docker Postgres container for each task, enable `vector` and `pg_trgm` before migrations, pass its URL through `GANTRY_TEST_DATABASE_URL`, and remove the container after the check. A plain `npm run test:integration` is allowed to skip those suites when the local Postgres test URL is absent.
- When adding DB-backed outbound delivery coverage, include the suite in the `test:integration:postgres` script so the required database gate cannot silently omit it.
- Claude Agent SDK boundary tests must stay hermetic: mock the SDK provider, assert generated options at the adapter boundary, and never require real Anthropic auth.
- Anthropic Claude execution lives under `apps/core/src/adapters/llm/anthropic-claude-agent/`; host spawn must use the provider-neutral execution adapter contract instead of naming Claude runner paths, config dirs, model env vars, SDK session options, or materializer helpers directly.
- Provider adapter registration must be explicit at bootstrap/composition; do not hide provider coupling behind concatenated dynamic imports from runtime or memory code.
- Execution adapters may project provider process metadata such as `modelCredentialEnv`, but must not patch host-owned authority or continuity fields such as allowed tools, selected MCP servers, session id, chat id, or thread id after runtime validation.
- Claude runtime materialization owns generated Claude config paths and protected-path projection; host spawn code should consume the execution adapter result instead of naming `.claude` settings or skills as durable inputs.
- Claude Agent SDK boundary env assertions must include `GANTRY_MEMORY_IPC_ACTIONS_JSON` in MCP server env (capability allowlist projection) and keep it out of top-level SDK process env.
- Control HTTP route changes must have route-level coverage for encoded ids, app ownership checks, and pre-mutation authorization.
- Control API bearer auth is JSON-only: use `GANTRY_CONTROL_API_KEYS_JSON` entries with explicit `kid`, `token`, `appId`, and `scopes`; do not reintroduce `GANTRY_CONTROL_API_KEY` or implicit all-scope local auth.
- Agent admin changes must show selected capabilities and bound conversation policies. The same agent can span Slack, Teams, Telegram, or Web, but sender policy, trigger policy, and approval authority stay on conversation ids.
- Persist canonical provider ids (`telegram`, `slack`, `teams`, `app`) in provider connections, messages, and participants. Short tokens such as `tg:` and `sl:` are JID prefixes only; normalize them before permission or membership checks.
- External ingress `session_message` dispatch must register the session group before enqueueing message checks; ingress and control adapters should share the same session interaction intent.
- External ingress target policy must mirror dispatch precedence: when `sessionId` is present, authorize only against `sessionIds`; use `conversationIds` only when no `sessionId` was supplied.
- SSE route writes that wait for backpressure must also unblock on response close/error and release subscriptions/active counters.
- Periodic external ingress retention and recovery sweeps must be bounded per timer tick; do not delete or update an unbounded backlog in one maintenance pass.
- MCP server changes must keep third-party servers behind approved Postgres definitions and agent bindings. Do not load `.mcp.json`, Claude user/project settings, raw stdio commands, or raw credential env as product truth.
- Third-party MCP materialization must fail closed for non-HTTPS/private/local URLs, remote hosts that resolve to private/link-local/loopback/multicast/metadata ranges, unsandboxed stdio templates, and missing Gantry Secret refs.
- Treat MCP `allowedToolPatterns` as an enforced allowlist, not metadata. Auto-approved MCP tools must remain a subset of the allowed tools, and same-channel rebinds must preserve any existing admin permission policies unless an admin explicitly replaces them.
- Agent-requested MCP credential needs are labels only. Never let the agent select arbitrary env keys; map them into server-scoped Gantry Secret refs before approval and materialization.
- MCP runner handoff files contain resolved credentials. Write them only after spawn preconditions pass and remove them in host cleanup paths; `npx-package` stdio templates may accept only one safe npm package argument.
- Resolved third-party MCP credentials must not be serialized into long-lived process env; use a private per-run handoff and keep SDK tool env sanitized.
- Host spawn must treat `selectedMcpServerIds` as the per-run third-party MCP allowlist. If the list is empty, do not resolve MCP Gantry Secrets or materialize active MCP bindings from storage.
- Broker model proxy values belong only in the model SDK credential lane. `NODE_EXTRA_CA_CERTS` may project only neutral TLS trust aliases (`SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `CURL_CA_BUNDLE`, `GIT_SSL_CAINFO`, `PIP_CERT`, `AWS_CA_BUNDLE`, `CARGO_HTTP_CAINFO`, and `DENO_CERT`) into the SDK process and approved Bash command prefix; keep provider tokens, broker proxies, browser env, and MCP stdio env out of tool subprocesses. `NO_PROXY` is compatibility only, not a safety boundary.
- Model-provider credentials are shared Model Access credentials. Resolve chat runs, subagents, memory, and jobs through the reserved `gantry-model-access` broker profile with `purpose=model_runtime`; do not add per-agent or `main-agent` fallback model credential bindings. Tool/API credentials must use `purpose=tool_capability` with explicit agent/capability context.
- Agent-facing broker failures must include a short cause, broker health/recovery guidance, and avoid leaking raw credential values; browser launch/status may expose broker health metadata only for broker-backed semantic capabilities.
- Browser is one durable capability: `Browser`. Runtime projects it into first-party Gantry MCP gateway tools: `mcp__gantry__browser_status`, `mcp__gantry__browser_open`, `mcp__gantry__browser_inspect`, `mcp__gantry__browser_act`, and `mcp__gantry__browser_close`; do not persist concrete browser subtool names as authority.
- Host spawn must fail closed on host-private browser backend allowed-tool rules and projected `mcp__gantry__browser_*` allowed-tool rules inherited from persistence; do not silently strip or canonicalize stale browser grants. Browser actions go through signed per-tool Browser IPC.
- Host-private browser backend MCP rules and projected Gantry browser gateway tools are stale or runtime-only names. Reject them anywhere persistent/readable agent tool rules are accepted, including settings, request_permission persistence, and inherited scheduler job policy; persist or select canonical `Browser` instead.
- Agent requests for browser automation through `request_skill_install` or `request_mcp_server` are wrong-lane requests; return guidance to request `toolName=Browser` via `request_permission` instead of recording a skill or third-party MCP review.
- `capability_status` must list unselected Browser as requestable with `permissionKind=tool toolName=Browser toolCategory=browser temporaryOnly=false`, while keeping Browser out of native SDK `availableTools`; selected Browser exposes first-party projected Gantry browser gateway tools, not raw backend MCP tools.
- Autonomous scheduler jobs resolve target-agent tools, skills, and MCP servers at run time. When a scheduled run hits a missing tool permission, use the same signed permission IPC path as interactive agents; `Allow once` resumes the current tool call, `Allow 5 min` creates the short timed grant, and `Always allow` updates the target agent capability binding, mirrors `settings.yaml`, and resumes the current run.
- Autonomous scheduler jobs activate only after setup readiness passes. Store declared `requiredTools`, `requiredMcpServers`, and structured `setup_state` on the job metadata path; keep `pause_reason` to `Setup required` and expose redacted blockers through API/CLI/MCP visibility instead of logs.
- Scheduled Browser job readiness must derive the Browser profile from the runtime group folder and conversation id, matching live run profile names such as `c-main_agent-...`; never derive it from canonical `agent:` ids or any external Chrome profile.
- Required-tool assertions are observability gates, not job-scoped grants. Validate them through the shared required-tool helper, publish `job.setup_required`/`job.tool_activity` evidence, and fail a `requiredTools: ["Browser"]` run explicitly when no browser IPC action occurs.
- Scheduled-job runner event forwarding must keep allowlisted `runtimeEvents` durable, including task notifications, permission lifecycle events, sandbox blockers, tool activity, and heartbeats. Do not regress the execution adapter to heartbeat-only forwarding.
- Scheduler jobs have no job-scoped extra tool grant surface. Browser access is requested and persisted as the canonical selected `Browser` agent capability, and the runtime projects it to `mcp__gantry__browser_*` tools when that capability is available for a run.
- Scheduler stale-lease cleanup must return the released job/run ids and route timed-out runs through terminal evidence delivery: runtime events, concise outcome notification, and `notified_at` after successful delivery. Do not let repository-only timeout updates be the final user-visible state.
- Browser job activity events for scheduled jobs must resolve the job's control app session before publishing; never publish app-owned job browser events under the default app unless the job has no session.
- Browser IPC polling must stay outside the main watcher hot path: hand off browser tool requests asynchronously with a bounded in-flight limit, keep backend clients lazily connected and closed on shutdown/profile close, and keep `runtime/ipc.ts` under the local line-count ratchet by moving owned subflows into runtime helper modules.
- Browser launch must not call the private resize backend. The headed launch size belongs in Chrome launch args such as `--window-size=1280,900`; explicit user-requested resize remains a separate browser IPC action after a page target exists.
- Browser launch must use an explicit nonzero loopback CDP port rather than `--remote-debugging-port=0`; Chrome exposes `navigator.webdriver=true` for port `0`. Do not use `--disable-blink-features=AutomationControlled` in the visible headed launch path because Chrome can show it as an unsupported command-line flag.
- Browser screenshot calls with `filename` must return a compact file reference, not inline base64 image content. Browser result hygiene must sanitize internal Chrome targets from every projected browser tool response because "Open tabs" blocks can appear on page-driving results, not only tab listings; target cleanup and launch flags are safety nets.
- Tool access inspection is single-cut: expose the canonical `toolAccess` object for jobs from inherited target-agent capabilities, and mirror persistent agent approvals to readable `agents.<id>.tools` settings entries.
- `settings.yaml` agent tools must be readable semantic capability ids, exact catalog tool names, or scoped Bash rules such as `capability:google.sheets.write`, `Browser`, `mcp__gantry__service_restart`, or `Bash(git status *)`. Reject scoped non-Bash rules and internal `tool:*` IDs in settings instead of rendering or reconciling them as compatibility state.
- Semantic tool capabilities persist as readable `capability:<id>` entries, such as `capability:google.sheets.write`; runtime may expand them to scoped Bash or adapter rules, but user prompts, API, MCP, and job recovery should keep the semantic capability name primary.
- User-defined local CLI capabilities must be reviewed through `propose_local_cli_capability` with pinned executable path/version/hash, scoped command templates, auth preflight, denied env overrides, and protected credential/config paths. They remain draft-only and must not project to runnable Bash until a runtime gate verifies those bindings per invocation. Never persist broad `Bash(cli *)` as a substitute.
- Agent capability truth must sync both ways: settings-side `agents.<id>.tools`, `skills`, and `mcp_servers` changes reconcile Postgres immediately by replacement, while Control API/admin-side capability writes must export readable projection back into `settings.yaml` before returning.
- Restart sync is mandatory for settings-owned agent/conversation state. Control API, CLI, and approved agent/admin-tool mutations to agent capabilities, provider connections, conversations, sender policies, control approvers, triggers, `requires_trigger`, or agent-conversation bindings must write the same nested shape to `settings.yaml` immediately; Postgres and live route rows are restart projections, not the only durable copy.
- App-owned scheduler jobs must authorize and resolve inherited tools through canonical `session_id`/control-session ownership only. Treat `linked_sessions` as delivery/routing metadata, never as app ownership truth or an app-id fallback.
- Live same-run tool approval rules are IPC security state. Store them under private directories/files and reject invalid run handles instead of normalizing or traversing paths.
- IPC response signing is per runner process. Every agent-originated IPC request that expects a signed response must carry the run's `responseKeyId`, and host responses must sign by that key id instead of looking up mutable agent/thread scope state; long-running calls such as `scheduler_wait_for_events` can outlive newer runs for the same conversation.
- Scoped scheduler job tool rules must be registered in `apps/core/src/shared/tool-rule-matcher.ts` with allow/deny tests before use. New native SDK tools are fail-closed until the evaluator knows the tool name and scoped input fields; MCP rules remain exact `mcp__server__tool` or wildcard `mcp__server__*`.
- Script-path Bash rules such as `Bash(/path/to/job.py *)` may match direct execution and safe Python interpreter execution (`python`/`python3 /path/to/job.py ...`), but must keep inline interpreters such as `python3 -c` and shell substitutions denied unless separately approved.
- Agent-facing scheduler MCP create/update tools must accept routing only through canonical `execution_context` and `notification_routes`; do not add top-level `linked_sessions`, `deliver_to`, `thread_id`, `session_id`, or `group_scope` aliases. Threads/topics are delivery metadata only and must not grant scheduler job visibility or run authority.
- Scheduler list paths must not pass `threadId: undefined` into repository filters; undefined means no thread filter, while `null` is the explicit unthreaded-only filter.
- Scheduler list/recent query predicates over `jobs.target_json` must stay index-aligned: use expression predicates for `executionContext.groupScope`/normalized `threadId` and containment predicates for `notificationRoutes` instead of per-row `jsonb_array_elements` scans.
- Provider session resume lookup is a hot path in `getAgentTurnContext`; keep `provider_sessions(agent_session_id, provider, status, updated_at DESC)` indexed when changing provider session schema or lookup predicates.
- Do not expose scheduler MCP list filters that the host will ignore. If an MCP tool is always scoped to the authenticated conversation, keep agent-facing schemas scoped the same way.
- `scheduler_wait_for_events` is a long-polling contract, not an immediate list alias. It must wait until matching events arrive or the requested timeout expires, and a five-minute wait must stay within the host IPC long-running path.
- Claude SDK `allowedTools` only auto-approves; it does not hide tools. Use runner-owned `tools` for available built-ins, `disallowedTools` for unsupported Claude Code built-ins, and `GANTRY_MCP_TOOL_NAMES_JSON` to keep duplicate or unselected Gantry MCP schemas out of agent context.
- Claude native SDK tools such as `Read`, `Write`, `Edit`, `Bash`, `Agent`, `WebFetch`, and `WebSearch` are Anthropic harness projections only. Do not seed or expose them as durable Gantry catalog capabilities; durable Gantry tools must be provider-neutral capabilities such as `Browser`, semantic `capability:<id>` entries, exact Gantry admin MCP tools, or scoped `Bash(...)` fallback rules.
- Owner-app job run/event listing must start from the app-owned control session route and join into jobs/runs/events; do not implement app scoping by scanning global recency pages and applying correlated ownership checks.
- Memory tuning knobs are settings-owned. If env keys are rejected by preflight, the exact replacement field must exist in `settings.yaml`, the runtime snapshot, defaults, renderer, validation/docs, and tests.
- Host runner sync code must work with npm workspace hoisting and installed package layouts; do not assume `packages/agent-runner/node_modules` exists.
- Files under `apps/core/src/app/bootstrap/` own composition and wiring only; runtime behavior must live in `runtime/`, `jobs/`, `session/`, `platform/`, `messaging/`, `memory/`, or infrastructure modules.
- Bootstrap helper splits must preserve existing architecture boundaries: keep helper modules dependency-injected and avoid introducing new direct adapter/channel imports in newly created bootstrap files.
- Channel provider catalog flags must match executable behavior: do not advertise `install` or `discover` unless the CLI/control path can actually perform setup or discovery, and document any remaining runtime adapter seam explicitly.
- Shared delivery error markers used by channel adapters and runtime/job settlement belong in provider-neutral domain modules, not under `runtime/`, so adapters do not import runtime internals.
- Permission approval suggestion synthesis must share the same validator as persistent permission writes. Bash is the intentional exception to generic exact-tool suggestions: durable Bash authority is parsed argv-leaf based, must not synthesize stateful leaves such as `cd`, and must not widen absolute path arguments into wildcard scopes.
- Native SDK tool availability is not authorization. The runner may expose permission-gated SDK tools such as `Bash`, `Write`, and `Edit` in `tools` so same-run approvals can bind, but `allowedTools`, `canUseTool`, live run-scoped permission rules, and scheduled-job policy remain the authoritative gates.
- YOLO-mode timed grants are not absolute when `permissions.yolo_mode.enabled` is true. Evaluate the shipped-plus-user denylist immediately before timed-grant auto-allow, emit an audit event on hits, and force the normal permission prompt instead of hard-denying.
- Tool execution policy is canonical in `apps/core/src/shared/tool-execution-policy-service.ts` so the standalone SDK runner can package it. Claude `PreToolUse`, SDK `canUseTool`, scheduler job rules, and browser/MCP projections must classify action target and mutation intent before allowing, denying, or requesting approval; do not add target-blind protected substring guards in runner adapters. Host-direct scheduler scripts remain disabled until they run inside an equivalent OS sandbox.
- `SandboxNetworkAccess` must stay an SDK-internal, short-lived run token check tied to an approved tool use with a matching parent tool-use id, or to the active same-agent/same-conversation eligible-tools/SDK-API-prompt timed grant. It is never a selected or persistent tool capability, and audit payloads must log only redacted tool-use ids, approved tool names, host hashes, input hashes, TTL/expiry, and the decision.
- Protected capability denial is target-based and fail-closed for shell commands that reference protected paths. Deny direct writes or shell access to `.mcp.json`, provider MCP/settings projections, skill capability files, symlinked protected targets, and `claude mcp add/remove/reset*`; only preserve explicit text-payload exceptions such as issue/PR bodies that mention those terms without targeting them.
- Autonomous scheduler recovery for missing tools must fail fast with the same request tools agents use, such as `capability_search`, `request_capability`, `propose_local_cli_capability`, `request_permission`, `request_skill_install`, `request_skill_proposal`, or `request_mcp_server`; approvals update the target agent capability bindings and `settings.yaml`.
- Permission approval timeout must stay human-scale by default (300000ms), resolve `PERMISSION_APPROVAL_TIMEOUT_MS` before `GANTRY_PERMISSION_TIMEOUT_MS`, and preserve runtime `.env` fallback for shared timeout consumers.
- Permission approval prompts and receipts must stay scan-readable and auditable: format tool inputs with typed sections, show long commands with head/tail truncation, keep persistent rules in Details/audit using safe rule summaries, and keep receipts tied to the exact approved action.
- Keep the architecture simple, do not over complicate
- Search Anthropic SDK in node modules and do not reinvent what already exists.
- User-triggered `/dream` in threaded Slack/Teams/Telegram topic routes must carry the active thread id into `runDreamingForGroup`/`triggerDreaming`; maintenance dedupe keys should include thread scope only for thread-scoped runs, while background/system dreaming remains group-scoped.
- Outbound delivery settlement must reject receipt rows whose `deliveryId`/`itemId` do not match the authenticated claim target, and recovery dispatch must always send to canonical delivery `conversationId`/`threadId` while treating provider destination hints as conflict checks only.
- Outbound recovery destination resolution must return routable provider-prefixed JIDs from canonical conversation metadata; when control-discovered conversations store raw external ids (for example Slack `C123`), derive the prefix from the provider connection and treat raw payload hints as equivalent to the canonical prefixed destination.
- Outbound recovery destination resolution and desired-state/discovery conversation ids must fail closed on explicit provider-prefix mismatches (for example `tg:` id on a Slack provider connection), and retry-tail provider metadata must pass through a strict allowlist sanitizer before any durable storage or API exposure.
- Cross-app outbound recovery claims must fail closed for external provider destinations when dispatch adapters cannot prove provider-connection identity; only internal `app:` session destinations may be recovered cross-app.
- Outbound recovery provider checks must normalize control-plane `control-http` app-session destinations to the internal `app` channel provider before comparing against channel wiring provider ids.
- Destination-hint normalization for outbound recovery must treat colon-containing hints with a non-alphabetic first segment (for example `19:...`) as raw provider ids and prefix them from the canonical destination before conflict checks.
- Channel wiring durable-send semantics are split: fail closed before provider send when `pending` persistence fails for required durability; after a visible provider send, if durable `sent` persistence fails, surface an ambiguous durable-delivery outcome, attempt to mark the row `partially_sent`, and never classify it as a provider failure or blindly retry the full payload.
- When a visible provider send succeeds but durable `settleSent` fails, immediately attempt a claim-token-authenticated durable `partially_delivered` settlement before throwing ambiguity; recovery and `claimDueDeliveryItems` must never blindly redispatch that potentially visible payload.
- Expired initial live-send first-attempt claims (reserved before provider dispatch) must be treated as ambiguous and non-auto-retryable; recovery claim paths must not blindly reclaim/redispatch them without proven provider idempotency evidence.
- Expired outbound `claimed` rows are non-retryable ambiguity by default in this repo: quarantine them as `partially_delivered` before claim selection, and only reclaim expired claims if provider-backed idempotency is explicitly implemented and tested.
- Expired outbound `claimed` row quarantine must stay bounded (`LIMIT` + `FOR UPDATE SKIP LOCKED`) and rely on the claimed-expiration partial index; never run an unbounded sweep before bounded claim selection.
- Retry-tail destination metadata for Slack `channelId` and Telegram `chatId` must be canonical-destination validated; mismatched ids must be omitted from persisted retry-tail payloads rather than trusted.
- Required channel sends must reserve durable outbound-delivery state (delivery + claimed item) before provider dispatch; message-row projections are not the gate for required-durability dispatch decisions.
- When a channel adapter throws `PartialMessageDeliveryError` with `retryTail`, persist that tail metadata durably and route suffix-only recovery through outbound delivery without creating nested legacy pending/sent rows.
- If retry-tail persistence for a visible partial send fails, surface an explicit incomplete/ambiguous durable-delivery outcome instead of implying recovery is available.
- Slack streaming fallback partial retries must persist `retryTail` as only the unsent fallback parts; never include already delivered fallback text.
- Telegram group stream final-edit failures after a visible head message must not resend the full rendered buffer; deliver only unsent overflow tails when possible, otherwise raise branded partial delivery with retry metadata only for provable unsent suffixes.
- Telegram MarkdownV2 escaping must treat `_`, `*`, and `~` as literal by default; preserve marker spans only when the caller explicitly marks formatting context.
- App channel runtime events must keep `canonicalText` metadata non-content and bounded (length/bytes/hash/content flags only), and never include plaintext previews or a second full copy of the event `text` payload.
- `ChannelWiring.sendProviderMessage` is recovery-only: recovery paths must mint a channel-wiring opaque permit bound to the claimed outbound item/destination before dispatch, and non-recovery files must not call `createRecoveryDispatchPermit` or `sendProviderMessage`.
- Runtime provider-visible text streaming may send live deltas only after passing through the stateful user-visible stream sanitizer. Never send raw provider deltas directly; final streaming calls close the generation and flush sanitizer carry.
- Runtime progress/status/receipt dispatch may use provider progress sinks, but progress text must stay host-authored status only. Do not send model/provider deltas, provider session handles, or arbitrary assistant text through progress updates.
- Runtime progress handles are turn-scoped by generation. Permission waits must pause active elapsed time and heartbeat updates, resume the same generation on prompt approval, and allocate a fresh generation only after background demotion or a new user turn.
- Runtime progress generation must stay separate from streaming-chunk generation. SDK success markers may advance streaming generations within a turn, but final progress such as `Done in N` must keep the active progress handle generation so follow-up receipts can be replaced.
- Runtime final progress such as `Done in N` must not be replace-only. Fast streaming can cancel the delayed initial progress before a handle exists, and terminal progress must still be allowed to create a final status.
- Runtime progress heartbeat elapsed-warning timestamps must initialize from the active turn start. Before starting a new queue turn, cancel any prior turn UI timers for that queue instead of relying on per-invocation cleanup to win races.
- Runtime shutdown must signal active interactive message runners with the normal IPC `_close` sentinel before waiting. Idle follow-up runners should exit through finalization so launchd restarts do not orphan stale progress handles or break provider-session continuity.
- Postgres expression indexes that include string constants must cast those literals explicitly (for example `'light'::text`) so fresh migrations do not fail with `data type unknown has no default operator class for access method btree`.
- Provider SDK session resume keys must be exact to the trusted runtime boundary: agent folder plus conversation JID, DM user id when applicable, and child thread/topic id. Never resume a provider `externalSessionId` from agent-folder-only state across DMs, channels, or sibling threads.
- Unsupported legacy continuity rows must fail closed; do not add runtime imports, backfills, or repair branches that turn old session ids or metadata-derived scopes into current `AgentSession` continuity.
- Runner logs must redact provider resume handles by field name, not only by value shape. Treat `sessionId`, `newSessionId`, `providerSessionId`, `externalSessionId`, `latestProviderSessionId`, and `session_id` as sensitive in raw stdout/stderr text, framed JSON, and structured log context.
- Runtime exception logging must preserve redacted `Error` type, message, code, cause, and stack details; never let structured runtime logs collapse thrown errors to `{}`.
- Scheduler terminal completion surfaces must redact provider resume handles before side effects. Apply redaction before writing `completeJobRun` summaries, `pause_reason`, lifecycle/runtime event summaries, and terminal scheduler notifications.
- Runtime live group output must use bounded user-visible accumulators for provider-visible delivery, redact provider resume handles after full-segment buffering, before channel formatting/streaming/fallback delivery, and before appending transcript summaries. Do not retain or join unbounded raw streamed deltas for provider-visible output.
