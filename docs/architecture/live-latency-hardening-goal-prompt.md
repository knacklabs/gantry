Strategic launch plan and provider-boundary decision:
- Postgres remains the launch source of truth for live latency and correctness.
- `LISTEN/NOTIFY` is wakeup-only. Durable rows and replay are the authority.
- Do not add Redis, SQS, Kafka, Redis pub/sub, or a generic queue-provider/cache seam for this goal.
- Future Kafka/SQS/Redis Streams support is allowed only as a narrow dispatcher adapter behind `event_bus_outbox` after measured pressure or a concrete multi-service consumer exists.
- `runtime_events` remains observable-only and must not become a command bus.
- Live continuation, stop, close, and interaction authority stays in durable live-turn command state backed by `live_turns`, `run_leases`, and fencing.
- DeepAgents live conversation state must default to LangChain's official `@langchain/langgraph-checkpoint-postgres` `PostgresSaver` backed by Gantry Postgres. Do not use SQLite for Gantry live runtime state, do not invent a custom file-backed `MemorySaver` or replacement saver, and do not store/replay raw `{role,text}` transcript files as the continuity mechanism.
- Anthropic SDK live conversation state remains SDK-owned: use SDK `persistSession` + `resume`/captured session id semantics. Do not manually reconstruct or replay prior channel messages into Anthropic SDK sessions.
- Performance work is not limited to persistence. First-visible latency must account for skills, MCP client startup, tool discovery/listing, tool schema size, permission IPC setup, sandbox/tool-network setup, model gateway setup, prompt/cache-control handling, and provider adapter initialization.
- First-visible latency is a UX constraint, not a reason to delete agent capability. Gantry should answer quickly or acknowledge work quickly, then keep the user informed while richer agent features continue. If latency is the priority for a run or agent, narrow the selected MCP/tool/skill projection before first visible output; do not globally remove async subagents, human-in-the-loop, streaming, skills, profiles, interpreters, hooks, or other reviewed features.
- Full-feature mode remains the product default for capable agents. Low-latency mode is a projection/profile choice that keeps only the few selected tools, MCP servers, and skills needed for the first response, then may lazy-load additional reviewed capabilities after first visible output when the run has authority and the user-visible status makes ongoing work clear.
- Do not narrow this goal to async subagents or any single official feature. Every implementation slice must carry the complete documentation review backlog: continuity/checkpointing, MCP inventory and call audit, selected tool projection and ToolSearch, skill and prompt/config discovery, permissions/HITL, sandbox/backends, streaming/live UX, structured outputs, observability/cost/progress, SDK extension controls, DeepAgents extension controls, retries/timeouts/scaling, frontend/protocol adapters, and benchmark/verification evidence.
- Treat the official-documentation review as a single findings set, not a sequence of optional notes. Async subagents are one row in that set; they do not outrank persistence, skills, MCP, tools, permissions, sandboxing, streaming/rejoin UX, structured outputs, telemetry, lifecycle/scaling, or extension-surface hardening.
- Official documentation findings belong in the findings register before implementation. Each slice must classify the relevant rows with disposition, evidence, owner/reason, and fail-closed behavior instead of silently omitting them.
- Required non-subagent parity: any update that adds, decomposes, or implements async/subagent work must also carry these peer findings forward in the same task notes: Anthropic SDK sessions/resume/fork/sessionStore, permissions/hooks, plugins/custom tools/MCP, skills and setting-source isolation, streaming input/output, structured outputs, slash commands, file checkpointing, todo/progress, cost and OpenTelemetry; DeepAgents skills, tools/MCP, HITL, streaming, profiles, interpreters, backends, sandboxes, memory/consolidation, context engineering, structured output, middleware, guardrails, production lifecycle, and frontend/protocol adapters; latest MCP tools/resources/prompts/roots/sampling/elicitation/tasks/auth/icons/schema metadata; LangChain frontend rejoin, message queues, headless tools, time travel, markdown streaming, tracing, tests, and evals. Treat SDK client lifecycle controls, built-in tool surfaces, MCP reconnect/toggle/status behavior, DeepAgents Code and managed-agent surfaces, LangChain MCP adapter behavior, model/profile switching, auth/config reload, direct skill invocation, startup shell commands, web/fetch tools, app-data hooks, and auto-update/install flows as explicit findings to classify, not implicit product affordances.
- Feature adoption and latency hardening are separate decisions. If a reviewed feature is not launch-now for a slice, classify it as `capability-gated`, `projection-disabled-for-low-latency`, or `deferred-until-reviewed` with fail-closed behavior and tests; do not omit it from the findings register because it is not being implemented in that slice.
- Use provider-native deferred-loading behavior where it exists. Anthropic SDK ToolSearch is the preferred large-tool-surface path when supported by the selected model/gateway; do not replace it with all-upfront tool schemas unless the tool set is small, the model/proxy does not support `tool_reference`, or the decision is measured and recorded.

Acceptance criteria:
1. Durable admission:
   - Add `live_admission_work_items` or equivalent durable runtime table.
   - Work rows are created atomically with inbound message persistence, or via a transactional outbox in the same commit.
   - Work rows reference persisted message id/cursor, app id, canonical conversation/thread/agent binding, sender identity, trigger decision metadata, idempotency key, state, retry/defer fields, claim worker/token/expiry, and fencing version.
   - Duplicate provider delivery creates exactly one work item.
   - Claiming is transactional and replay-safe using durable state, not notify payloads.
   - Capacity deferral is persisted, FIFO by original message time, de-duped for status emission, and retried on capacity release.
   - Retry/defer readiness ordering and stale-claim expiry ordering may bound the candidate window by `defer_until` or `claim_expires_at`, but final claim order should preserve original message time across the selected window and must not weaken capacity-deferral FIFO for rows waiting only on capacity.

2. Evented wake and replay:
   - Live admission normal path no longer scans all routes on `POLL_INTERVAL`.
   - `LISTEN/NOTIFY` carries only opaque ids or tiny metadata and never prompt text, provider tokens, conversation titles, or authority.
   - Dropped/coalesced NOTIFY recovers by durable replay without route-wide scans.
   - Recovery sweep remains for dead owners, not normal hot-path admission.

3. Live-turn commands and bridge:
   - Continuation, stop, close, and interaction resolution append durable commands first.
   - Owner wake drains by durable turn id + seq cursor.
   - Add a provider-neutral per-run evented bridge for Anthropic and DeepAgents live runs.
   - Bridge frames are authenticated with a per-run unguessable secret or HMAC and include run id, live turn id, lease token, fencing version, command seq, nonce/idempotency key, command type, scope, and payload.
   - Stale worker, stale fence, wrong run id, wrong thread, replayed nonce, and post-finalization frames are rejected.
   - Reconnect replays unacked commands by durable cursor.
