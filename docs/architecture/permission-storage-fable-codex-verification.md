# Permission Storage Fable — Codex Verification

Validated checkout: `main` at `88c288d20` (merged PR #229).

## Claim 1 — Duplicate claim equality has incompatible live semantics

**Verdict: CONFIRMED, with one path correction.**

- `samePersistedPermissionClaim` compares the complete serialized claim (`apps/core/src/application/interactions/pending-interaction-permission-claim.ts:60-65`). That includes `intent.decidedAt` and `match.providerAliases`, because both are fields of `PermissionCallbackClaim` (`apps/core/src/domain/types.ts:206-223`). Its live batch consistency check rejects the lookup when any row is missing a claim or is not serialization-identical (`apps/core/src/application/interactions/pending-interaction-permission-callback.ts:307-317`).
- `samePermissionCallbackClaim` compares fields through `match.canonicalId` and includes `intent.decidedAt`, but it deliberately does not compare `match.providerAliases` (`apps/core/src/application/interactions/pending-interaction-permission-envelope.ts:179-194`). Its live caller accepts such claims and then unions their aliases (`apps/core/src/application/interactions/pending-interaction-prompt-binding.ts:401-434`). The alleged Telegram path is stale: the comparator is called in the application prompt-binding file; Telegram reaches that application helper from `apps/core/src/channels/telegram/permission-callback.ts:96-102`.
- The SQL copies the input claim into every claimed row but overwrites `match.providerAliases` from that row's former `permissionCallbackId` (`apps/core/src/adapters/storage/postgres/repositories/worker-coordination-interaction.postgres.ts:417-432`). The focused Postgres test creates two batch members with different aliases, proves that each stored claim contains its own alias, and then expects request-id recovery to return `null` (`apps/core/test/integration/worker-coordination.postgres.integration.test.ts:946-1051`).

Concrete failure: batch members R1 and R2 share batch scope B but have provider aliases A1 and A2. One callback wins claim C; the single SQL update writes otherwise-identical C copies with aliases `[A1]` and `[A2]`. If the host dies after that claim and before settlement, prompt-message recovery can accept the copies and union their aliases, while request-id recovery compares the whole serialized claims and returns `null`; Telegram then fails its durable lookup at `apps/core/src/channels/telegram/permission-callback.ts:112-125` (and Slack's restart branch likewise depends on that lookup at `apps/core/src/channels/slack/channel-interactions.ts:303-317`). This is a real recovery landmine, not only an object-key-order concern.

## Claim 2 — A single envelope row is safe

**Verdict: PARTIAL.** One canonical envelope row is sound, but the exact proposal — one envelope-level claim plus members carrying only `{envelope_id, index, status}` and an unqualified `UPDATE ... WHERE claim_id IS NULL` — is not behavior-preserving.

Breaking cases:

1. **Live Review each needs both batch settlement and member claim state.** A batch `allow_persistent_rule` decision means `review_each`, not a grant for all members (`apps/core/src/channels/permission-batch-coalescer.ts:59-78`). The live path settles the batch claim and then obtains a distinct decision for every member (`apps/core/src/channels/permission-approval-requester.ts:217-238`); rebinding an individual member deliberately preserves the prior batch settlement while replacing batch callback state with member callback state (`apps/core/src/application/interactions/pending-interaction-prompt-binding.ts:185-205`). The current claim SQL also records the transition by flipping the batch phase (`apps/core/src/adapters/storage/postgres/repositories/worker-coordination-interaction.postgres.ts:433-445`). One envelope `claim_id/settled_at` cannot simultaneously retain the terminal batch owner and fence the subsequent per-member claims. A corrected one-envelope design therefore still needs member-scoped claim/settlement ownership or a separate claims table; deleting cross-restart replay under Claim 3 removes the durable replay phase, but it does not remove this live two-level claim invariant.

2. **The envelope CAS must retain exact-membership and per-member-expiry guards.** Today a batch is claimable only when the exact and distinct marked-member counts equal the declared batch length and every counted member is pending and has `expires_at > decidedAt` (`apps/core/src/adapters/storage/postgres/repositories/worker-coordination-interaction.postgres.ts:344-396`). The final update repeats the pending/unexpired predicate and exact batch scope (`apps/core/src/adapters/storage/postgres/repositories/worker-coordination-interaction.postgres.ts:447-464`). `UPDATE permission_prompts SET claim_id = ... WHERE claim_id IS NULL` alone can win after a member expires, resolves, or disappears. Per-member expiry can survive if members remain backed by `pending_interactions.expires_at` (`apps/core/src/adapters/storage/postgres/schema/worker-coordination.ts:135-166`) and the envelope claim uses a relational `NOT EXISTS`/count guard over those rows; if the proposed member rows literally retain only the three listed fields, expiry cannot be enforced.

3. **Question reopen and lease-liveness behavior: none found, provided the cutover is permission-only.** Question creation updates on idempotency conflict only when the existing row is both `kind='question'` and `status='cancelled'` (`apps/core/src/adapters/storage/postgres/repositories/worker-coordination-interaction.postgres.ts:71-105`). Lease-loss cancellation is separately fenced to a pending question with a concrete run and an inactive exact lease token/fencing version (`apps/core/src/adapters/storage/postgres/repositories/worker-coordination-interaction.postgres.ts:244-285`). A separate `permission_prompts` table and permission-only joins do not conflict with either rule; a generic inner join or shared conflict rewrite would, so those question queries must remain outside the schema cutover.

## Claim 3 — Cross-restart Review each replay can be deleted

**Verdict: CONFIRMED, with an atomic deletion boundary.**

The exact source-consumer inventory is closed:

- New batch envelopes start at `phase: 'decision'` (`apps/core/src/application/interactions/pending-interaction-prompt-binding.ts:136-153`), and envelope parsing accepts only `decision | review_each` (`apps/core/src/application/interactions/pending-interaction-permission-envelope.ts:96-105`).
- The only production writer of `review_each` is the batch persistent-mode claim update (`apps/core/src/adapters/storage/postgres/repositories/worker-coordination-interaction.postgres.ts:433-445`).
- The only production readers are the request replay branch (`apps/core/src/application/interactions/pending-interaction-permission-callback.ts:111-137`), `replayPersistedReviewEach` itself (`apps/core/src/application/interactions/pending-interaction-permission-callback.ts:151-234`), and the durable post-restart resolver (`apps/core/src/application/interactions/pending-interaction-permission-callback.ts:539-553`).
- `reviewEachReplays` is only the in-process promise memo for that helper and is otherwise set, cleared, or configured at `apps/core/src/application/interactions/pending-interaction-permission-callback.ts:161-168`, `:231-265`; its dispatcher is registered at `apps/core/src/channels/permission-approval-requester.ts:319`. The normal live Review each path uses its own direct member loop (`apps/core/src/channels/permission-approval-requester.ts:217-238`), not the replay memo.

Deleting the phase, SQL phase flip, replay helper/memo, recovery dispatcher registration, and their replay tests therefore does not orphan unrelated production behavior if restart now terminalizes/inactivates that unfinished batch and lets the agent re-ask. The cut must be atomic: a settled batch `allow_persistent_rule` claim must not fall through to `recoveredPermissionDecision`; the old review-each rows must first become terminal/ineligible for plain replay. Keep `replayPersistedPermissionDecisionForRequest` for non-review-each claims: it still reconstructs ordinary decisions from the persisted request and claim (`apps/core/src/application/interactions/pending-interaction-permission-callback.ts:51-141`, `:667-680`), and the IPC path consults it before opening a new prompt (`apps/core/src/runtime/ipc-interaction-processing.ts:144-194`). Individual `allow_persistent_rule` and reviewed semantic-capability approvals continue through the generic persistent-grant path (`apps/core/src/domain/permission-decision.ts:91-135`, `apps/core/src/application/interactions/pending-interaction-grants.ts:39-63`), which rechecks paused jobs and queues ready ones (`apps/core/src/application/interactions/pending-interaction-permission-recovery.ts:46-103`, `apps/core/src/application/jobs/job-permission-recovery.ts:58-101`). Batch `allow_persistent_rule` is the Review each control action and is intentionally the exception that re-asks after restart.

## Reconciliation

Choose **B: characterize-and-sweep dead restart replay → unify the permission recovery orchestrator on the current repository port → land one merged envelope/member/claim schema cutover**. Claim 3 removes phase/replay state first, so the new schema should not institutionalize state the product has chosen to drop. Claim 1 is a live semantic split that one orchestrator can eliminate once before the representation migration. Claim 2's exact-membership, expiry, terminal-owner, and live batch/member-claim invariants must be characterized during the sweep, then migrated behind that single orchestrator rather than through four provider flows.
