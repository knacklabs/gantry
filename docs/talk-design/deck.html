I'll be straight with you before I go further. MyClaw's code today still reads the OAuth token env var. I wrote that code before the policy enforcement, and I keep it because the SDK supports it. MyClaw is a personal tool. I'm transparent about that. I wouldn't ship it to a company as a subscription-evasion layer — I'd feel bad about that. I don't. What I built is an answer to the question: how do I run a serious Claude-based personal assistant on the API path, which is the supported path, without paying two thousand dollars a month? So I built MyClaw
  • Personal.

So in February and March, instead of fighting the policy, I sat down and rebuilt the thing. MyClaw. Personal. Small. Built with one goal: make API-path usage sustainable for a single-user personal assistant. That's not a feature list. That's a design constraint. Every choice downstream of that constraint — the SDK I picked, the memory architecture, the permission model — follows from trying to use fewer tokens per turn, cache more aggressively, and reach for Opus only when it actually matters. Closer to the primitive
@anthropic-ai/claude-agent-sdk

Not a wrapper.
Not a custom loop.
The SDK itself.

Every group gets its own CLAUDE.md and SOUL.md. Skills, subagents, the permission hook — all included.

Open packages/agent-runner/src/index.ts on your laptop. Scroll to line 508. This is the agent runner. The whole thing. It's a call to query() from the Claude Agent SDK, and a streaming loop around the response. I don't own the agent loop. I don't own the tool dispatcher. I don't own the skill system. All of that comes from @anthropic-ai/claude-agent-sdk. OpenClaw and Hermes are both on the raw Anthropic SDK — the completion API. That means they wrote their own loop, their own tool parsing, their own skill system, their own approval layer. Good engineering. Not what I wanted to maintain. Because I'm on the Agent SDK, two things come essentially for free. ONE: every group has a home. Look at apps/core/src/runtime/prompt-profile.ts:14. Every Telegram chat, every Slack channel gets its own directory at ~/myclaw/agents//. Inside: a CLAUDE.md the agent reads every turn, a SOUL.md that holds the group's identity. The agent's working directory IS the group's directory. If you've used Claude Code in a repo, this is the same pattern — now every chat is a repo. TWO: prompt caching. Because the Agent SDK uses Claude Code's system prompt preset and because the group CLAUDE.md files are static, a large fraction of every turn is cache-hit. That is the single biggest reason API usage is affordable at all. Without caching, none of the rest of the economics work. live demo
DO THE DEMO FIRST. 60 seconds. "Watch. I'm going to ask the agent on my laptop to write a file. Anyone who's used these tools knows the agent just writes it. But…" Trigger the agent. Your phone lights up. Hold the phone up to the room. Tap approve. The agent writes the file. "Okay. Now let me tell you what just happened. The Claude Agent SDK has a hook called canUseTool. Before any tool fires, the SDK asks: 'agent wants to call this tool with these arguments — yes or no?' You can return true, false, or prompt for an answer. Most code uses this hook to block a handful of dangerous commands. MyClaw uses it for every tool call, by default. When the hook fires, MyClaw writes a permission-request file to an IPC directory. The host watches the directory — rate-limited, auth-token-validated — and renders the request as a Telegram inline keyboard or a Slack interactive block. Your reply flows back through the same IPC channel. The hook returns, the tool runs. Total round-trip: usually under two seconds. If you don't reply in five minutes, default is deny. Here's the interesting part. I checked both OpenClaw and Hermes. Neither uses canUseTool in their runtime — not one match. They built their own approval systems from scratch because the raw Anthropic SDK doesn't have this hook. Doing that for just bash is reasonable. Doing it for every tool would be a substantial rewrite. MyClaw gets it free because it sits on the SDK that ships with it. There's also a separate tool called ask_user_question for multi-choice prompts — 'draft this formal or casual?' Same IPC plumbing, buttons in Telegram, modals in Slack. SQLite + sqlite-vec · scheduled consolidation

Last piece. Memory. Memory in MyClaw is a SQLite database per group, with the sqlite-vec extension for vector search. No Redis. No Pinecone. The retrieval query isn't just cosine similarity — it's reciprocal-rank fusion of BM25 lexical search and vector similarity, with temporal decay (newer memories outrank older ones) and maximal marginal relevance to deduplicate near-identical results. Four standard IR techniques. Combined, they produce a ranked list that's meaningfully better than any one of them alone. The top results go into a hidden message block that prepends the user prompt. The agent sees it as context. You never see the mechanism. The interesting piece is what happens on a schedule. MyClaw has a persistent cron scheduler. One of the jobs is memory DREAMING. It runs on a schedule, reads raw memory chunks from recent conversations, groups them, rewrites them as higher-level items — facts, procedures, preferences — and puts them back in the same SQLite. Think about that for a second. The agent's memory is being rewritten by the agent itself, on a schedule, with no human in the loop. It's consolidation. Like what your brain does during sleep. Hence 'dreaming'. Over weeks, the raw noise fades and the useful abstractions stay. Why does this matter for the economics? Because consolidated memory means shorter effective context at inference time, which means fewer tokens per turn, which means API usage stays affordable. Every piece is in service of the same constraint.
What I gave up
  • Locked in to Anthropic. No provider swap.
  • Two channels: Telegram, Slack.
  • IPC is filesystem polling. Debuggable. Not free at scale.

It's a personal tool. That's the point.

A few things I gave up. One: provider portability. MyClaw runs on Claude. If I want to swap to another provider, I'm writing a new runner. Hermes can flip a config line. I can't. Two: channel breadth. Two channels today — Telegram and Slack. If you need iMessage, Signal, Discord, there are better tools. Three: the IPC layer. Every tool call writes a JSON file. Every response reads one. It's unreasonably debuggable — I can tail -f the IPC directory and watch the agent think. But it's not free at scale. Single-user, small-team — it's fine. A public service with thousands of users would need a real transport. This is a personal tool. It's built for my constraints. It probably fits some of yours. If not, use OpenClaw or Hermes — they're the right answer for different constraints. (Do not say "MyClaw is ahead of them" or "better than them." The story already made your case.
